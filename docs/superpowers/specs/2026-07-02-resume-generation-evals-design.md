# Résumé-generation evals — golden dataset + LangFuse LLM-judge

- **Date:** 2026-07-02
- **Status:** Approved design — ready for implementation plan
- **Area:** `dashboard/` (Next.js résumé pipeline + UI), Postgres, LangFuse

## Problem

We evaluate the job **reviewer** both offline (a human-corrected golden dataset
replayed through `reviewer/experiments.py`) and online (LangFuse traces with
per-field scores). We have **no equivalent for résumé generation.** Tailored
résumés are generated by `generateResume()` (`dashboard/lib/rolefit/`) and the
call already emits a `resume-generation` LangFuse span, but nothing scores that
output. We can't tell whether a prompt or model change makes résumés better or
worse, and we have no ground truth to measure against.

The reviewer's pattern can't be copied verbatim: the reviewer emits categoricals
and 0–100 sub-scores with deterministic ground truth (exact-match / tolerance
evaluators). A résumé is **free text** — the tailored summary, skills, and
per-role bullets have no single correct answer. Quality has to come from human
judgment (offline) and an LLM-as-judge (online), scored against a shared rubric.

## Goal

1. **Offline golden dataset.** Let the operator score real generated résumés
   (grounding + JD-relevance, 1–5, plus a free-text comment) in the dashboard,
   persist the scores, and sync them to a LangFuse `resume-golden` dataset.
2. **Online LLM-judge.** Make production résumé traces evaluator-ready so a
   **LangFuse-managed** LLM-as-judge scores every generation on the same rubric.
3. **Calibration loop.** Because the human score and the judge score attach to
   the *same* résumé (via its trace id), report judge-vs-human agreement so we
   know whether to trust the online judge.
4. **Deterministic mechanical checks** on every generation (length/one-page fit,
   verb repetition, skills dedup, roles-present, anti-hallucination), attached to
   the trace as in-code scores and surfaced to the operator while scoring.
5. **Regression plumbing in place** so a later "re-generate the golden inputs
   under a new model/prompt → judge → compare" harness is a drop-in follow-up.

## Non-goals

- The regression **harness itself** (re-generation over golden inputs as a
  LangFuse dataset run). Only its plumbing lands now; the runner is a follow-up.
- Language-quality and impact/specificity judge dimensions — deferred by choice;
  the rubric is grounding + JD-relevance only.
- Multi-operator / multi-labeler workflows — single operator (`isAuthed`), like
  the reviewer corrections tool.
- Changing the résumé generation prompt or model. Evals observe; they don't
  alter the pipeline.

## Approach decisions

Settled during brainstorming:

1. **Shared rubric + calibration.** One rubric (two dimensions) powers both the
   offline human scoring and the online judge; a calibration step measures their
   agreement.
2. **Two judge dimensions, 1–5:** `grounding` (every claim traceable to the
   candidate's real background — the #1 risk) and `jd_relevance` (content
   selected/emphasized toward the target role). Overall = grounding-weighted mean
   (grounding 0.7 / jd_relevance 0.3 — fabrication is the dominant failure).
   Weights are a constant, tunable later.
3. **Online judge = LangFuse-managed evaluator** (configured in the LangFuse UI),
   for near-zero online code. The rubric prompt is **mirrored in-repo as a
   reference constant** so it's versioned even though LangFuse holds the running
   copy.
4. **Golden = production résumés,** scored inline in the board. Each maps to a
   live trace the managed judge already scored, so calibration is a direct
   human-vs-judge join by `resume_trace_id` — no re-generation needed.
5. **Mechanical checks are in-code** (deterministic, cheap, exact); the subjective
   axes are the only thing the LLM-judge rates. Clean split of responsibilities.
6. **Push-on-save**, best-effort, DB-first — mirrors `saveReviewCorrection`: the
   Postgres write is durable before the LangFuse dataset upsert; a failed push is
   reconcilable via a backfill.

## Scoring model

| Signal | Source | Where it lives | Scale |
|--------|--------|----------------|-------|
| `grounding` | human (offline) + managed judge (online) | golden dataset `expectedOutput` (human); trace score (judge) | 1–5 |
| `jd_relevance` | human (offline) + managed judge (online) | golden dataset `expectedOutput` (human); trace score (judge) | 1–5 |
| `overall` | derived | `0.7*grounding + 0.3*jd_relevance` | 1–5 |
| mechanical checks | deterministic code | trace scores + `/api/resume` response | bool / count |

## Architecture & data flow

```
Generate résumé (/api/resume, /api/application/prepare)
   │  ┌─ parent "resume" observation: input={title,company,description,background}, output=résumé text
   │  ├─ generation span "resume-generation" (existing: prompt in, TailoredResume out, cost)
   │  ├─ generateResume also returns resumeChecks(resume, parsedProfile) ── route attaches them as trace scores
   │  └─ capture LangFuse trace id ── persist application_packages.resume_trace_id
   ▼
LangFuse-managed judge (UI-configured) ── scores the trace: grounding, jd_relevance (1–5)
   ▼
Operator opens ResumePanel ── "Score résumé" ── grounding 1–5, jd_relevance 1–5, comment
                                    │ (mechanical checks shown alongside to guide scoring)
                                    │ Save
                                    ▼
              saveResumeScore(jobId, scores)  [server action]
                 1. requireUserId()
                 2. SELECT résumé snapshot + trace id + gen inputs (application_packages, jobs, profiles)
                 3. UPSERT resume_scores  (grounding, jd_relevance, comment, resume_trace_id, snapshot, model)
                 4. upsert LangFuse resume-golden dataset item  (@langfuse/client)  ← best-effort
                 5. revalidatePath("/")
                                    │
   (terminal) node scripts/calibrate-resume-judge.ts
                 for each golden item: fetch judge scores on resume_trace_id,
                 compare to human scores → per-dimension agreement report
                                    │
   (future)  node scripts/run-resume-experiment.ts  ← plumbing ready now
                 re-generate over golden inputs → LangFuse dataset run → managed judge scores it
```

The DB is the source of truth; LangFuse is kept live on save and reconcilable via
a `--sync` backfill.

## Component 1 — Traces evaluator-ready (online; mostly already there)

`callOpenRouterStructured` already opens a `resume-generation` generation span
(prompt in, `TailoredResume` out, usage + real OpenRouter cost). Two additions
make the trace a clean, unambiguous target for a UI-configured evaluator:

**Parent observation + clean I/O — `/api/resume/route.ts`, `/api/application/prepare/route.ts`**
Wrap the generation in a parent observation named `resume` whose `input` is the
structured generation context (`{title, company, description, background}`) and
whose `output` is the rendered résumé text (`composeResumeText`). The managed
evaluator maps its `{job_description}` / `{resume}` variables from this parent,
rather than parsing the raw system+user prompt off the generation span.

**Trace-id capture**
Read the active LangFuse/OTel trace id inside `run` and thread it back so the
route can persist it on the application package (`resume_trace_id`). Exact
accessor (`@langfuse/tracing` current-observation helper vs. the OTel
`trace.getActiveSpan().spanContext().traceId`) is verified at implementation
time; the value is a hex trace id string.

**Managed evaluator (LangFuse UI, documented in-repo)**
- New reference constant `dashboard/lib/rolefit/resumeJudgeRubric.ts`: the exact
  judge system prompt + the two-dimension 1–5 rubric + the JSON output shape
  (`{grounding, jd_relevance}`), with a header comment stating LangFuse holds the
  running copy and this file is the versioned source. A drift note like
  `taxonomy.ts` carries.
- Configure the evaluator in LangFuse to run on résumé traces (filter on the
  `resume` observation / a `resume` tag), **judge model = Claude Sonnet 5** (the
  exact LangFuse LLM-connection slug — e.g. `anthropic/claude-sonnet-5` on
  OpenRouter, or the Anthropic connection's model id — confirmed against the
  current model list when wiring the evaluator), mapping `job_description` and
  `resume` from the parent I/O. **Also enable it on dataset-run outputs** (not
  only live traces) so the future regression harness is auto-scored.
- Sampling: 100% online at current volume; overridable via the existing
  `LANGFUSE_SAMPLE_RATE`.

## Component 2 — Human scoring UI (offline golden)

**Scoring panel — `dashboard/components/rolefit/ResumeScorePanel.tsx` (new)**
Rendered inside `ResumePanel`/`ApplicationPanel` when `isAuthed` and a résumé
exists. A **"Score résumé"** button toggles an edit sub-mode:
- `grounding` 1–5 and `jd_relevance` 1–5 (segmented control or 5-button row),
  a `comment` textarea, **Save** / **Cancel**.
- The **mechanical check results** for the current résumé render read-only above
  the inputs (green pass / red fail with the offending detail), so the operator
  scores with the deterministic signals in view.
- Save calls `onSaveScore(jobId, { grounding, jd_relevance, comment })`; a
  non-fatal toast surfaces a LangFuse-sync failure ("Saved. LangFuse sync failed
  — will reconcile."). Pre-fills from an existing `resume_scores` row so a score
  is editable.

**Server action — `dashboard/app/actions/resumeScores.ts` (new)**
`saveResumeScore(jobId, scores)`:
1. `requireUserId()`.
2. `SELECT` the current `application_packages` row (résumé snapshot, model,
   `resume_trace_id`) and the generation inputs (`jobs` title/company/description;
   `profiles` résumé text) for the dataset item.
3. `UPSERT resume_scores` (`ON CONFLICT (user_id, job_id) DO UPDATE`), storing
   the résumé snapshot + trace id + model for provenance.
4. Best-effort upsert the LangFuse dataset item via `lib/resumeGoldenDataset.ts`
   (DB commit already durable). Return `{ ok, langfuseSynced }`.
5. `revalidatePath("/")`.

## Component 3 — Data model

**New table `resume_scores` (migration `migrations/2026-07-02-resume-scores.sql`, mirrored into `schema.sql`)**
Keyed `(user_id, job_id)` — one score per résumé per operator, re-scoring
overwrites (matches `application_packages`' `UNIQUE (user_id, job_id)`).

```sql
CREATE TABLE resume_scores (
  user_id          UUID NOT NULL,
  job_id           TEXT NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
  grounding        INT  CHECK (grounding    BETWEEN 1 AND 5),
  jd_relevance     INT  CHECK (jd_relevance BETWEEN 1 AND 5),
  comment          TEXT,
  resume_trace_id  TEXT,                                -- join key to the managed judge's trace score
  resume_snapshot  JSONB NOT NULL DEFAULT '{}'::jsonb,  -- the exact TailoredResume scored (survives regeneration)
  model            TEXT,                                -- model that generated the scored résumé
  scored_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, job_id)
);
CREATE INDEX idx_resume_scores_user ON resume_scores (user_id);
```

**New column** `application_packages.resume_trace_id TEXT` (nullable) — the
LangFuse trace id captured at generation, so a later score can reference the
judge's trace even if the résumé is regenerated (`resume_scores.resume_snapshot`
pins exactly what was scored).

RLS: deny-all, following the `2026-06-26-rls-deny-all-policies.sql` convention
(access is via the service-role DIRECT connection; nothing served through the
anon API).

## Component 4 — LangFuse dataset + calibration

**Dataset helper — `dashboard/lib/resumeGoldenDataset.ts` (new)**
Wraps `@langfuse/client` (already installed; same pattern as `langfuseDataset.ts`)
to upsert one `resume-golden` item:
- `id = ${userId}:${jobId}` (deterministic → re-scores upsert).
- `input` = the **full generation inputs** (`{title, company, description,
  background, model}`) — enough to re-generate for regression.
- `expectedOutput` = `{ grounding, jd_relevance, comment }` (human ground truth).
- `metadata` = `{ resume_trace_id, model, scored_at, source: "dashboard" }`.

Exact SDK method verified at implementation; payload shape is stable.

**Calibration + backfill — `dashboard/scripts/calibrate-resume-judge.ts` (new)**
- Default mode: for each `resume-golden` item, fetch the managed judge's scores
  on its `resume_trace_id` (LangFuse scores API), compare to the human
  `expectedOutput`, and print a per-dimension report — exact-agreement rate and
  mean absolute error for `grounding` and `jd_relevance`, plus a list of the
  largest disagreements (item id + human vs. judge). This is the "should I trust
  the online judge" signal.
- `--sync` mode: re-push every `resume_scores` row to the dataset (reconcile if a
  live push failed). Mirrors `python -m reviewer.experiments sync`.

## Component 5 — Deterministic mechanical checks

**Pure module — `dashboard/lib/rolefit/resumeChecks.ts` (new)**
`resumeChecks(resume: TailoredResume, parsed: ParsedProfile)` → a structured
result of pass/fail + counts, derived from the generation contract in
`resumeSchema.ts`:
- **Anti-hallucination:** every `experience[].company` in the output matches a
  parsed-profile company; every parsed role is present and **in order**.
- **Volume / one-page:** per-role bullet counts in range (2–7), skills count
  12–16, summary ≤ 70 words, headline ≤ 55 chars, each bullet ≤ 24 words; a
  total-content heuristic for one-page fit.
- **Repetition / dedup:** no repeated opening verb across bullets; no duplicate
  or subsuming skills (e.g. both "AWS" and "AWS S3").

**Computed where the data is:** `generateResume` (via `resumeClient.ts`) already
holds the assembled `TailoredResume` and the `ParsedProfile`, so it runs
`resumeChecks` and returns them alongside the résumé (`{ resume, checks }`) —
routes never re-parse the profile. Both générating routes (`/api/resume` and
`/api/application/prepare`) then (a) attach the checks to the trace as in-code
numeric/boolean scores via `@langfuse/tracing`, and (b) include `checks` in their
JSON response so the scoring panel renders them for whichever résumé is on screen.
Pure functions → straightforward unit tests over crafted `TailoredResume` /
`ParsedProfile` fixtures.

## Regression plumbing (in place now; harness deferred)

Nothing runs regression yet, but the follow-up is a drop-in because:
1. The `resume-golden` `input` carries the full generation inputs → a run can
   re-generate deterministically from a dataset item.
2. The managed evaluator is enabled on **dataset-run outputs**, so a regression
   run is judged with zero extra wiring.
3. `model` is recorded in item metadata → runs are comparable across models.
4. `generateResume(...)` is already script-callable with explicit inputs (see
   `scripts/gen-resume.ts`), so `scripts/run-resume-experiment.ts` is just:
   dataset items → `generateResume` → LangFuse dataset run.

## Error handling

- DB write commits before the LangFuse push; a score is never lost to a sync
  failure. Failed pushes reconcile via `calibrate-resume-judge.ts --sync`.
- LangFuse push failures are caught, logged, returned as `langfuseSynced: false`
  → non-fatal toast.
- Trace-id capture is best-effort: if tracing is disabled or the id is
  unavailable, `resume_trace_id` is null; the score still saves and the golden
  item still syncs (calibration simply skips items without a judge trace).
- Mechanical checks never throw on odd input — a missing section yields a failed
  check, not an exception; generation is never blocked by the checks.
- Last-write-wins on the `(user_id, job_id)` upsert; single operator.

## Testing

**Dashboard (vitest)**
- `resumeChecks` — pure-function tests: anti-hallucination (injected foreign
  company fails; correct order passes), volume boundaries (skills 11/12/16/17,
  summary word count, bullet length), verb-repeat and skills-dedup detection.
- `saveResumeScore` — writes the row, builds the correct dataset payload,
  tolerates a LangFuse-sync failure (returns `langfuseSynced: false`, still ok).
- `resumeGoldenDataset` — the item-payload builder (id, input, expectedOutput,
  metadata) from a fixture score + inputs.

**Manual live pass**
Set `LANGFUSE_*` + `OPENROUTER_API_KEY`, `npm run dev`, sign in. Generate a
résumé → confirm a `resume` trace appears in LangFuse (**US region** —
us.cloud.langfuse.com) carrying the managed judge's `grounding` / `jd_relevance`
scores and the mechanical check scores. Score the résumé in the UI → confirm a
`resume-golden` dataset item with the human `expectedOutput`. Run
`node scripts/calibrate-resume-judge.ts` → confirm the agreement report joins the
two.

## Rollout / deploy

Order matters (migration-coupled code):
1. Apply `migrations/2026-07-02-resume-scores.sql` (new table + the
   `application_packages.resume_trace_id` column) to Supabase **before** pushing
   code that reads/writes them.
2. Confirm `LANGFUSE_*` env vars on the Vercel project (Production + Preview) —
   the reviewer-golden / résumé-tracing work already introduced these.
3. Configure the managed evaluator in the LangFuse UI (rubric from
   `resumeJudgeRubric.ts`); enable it on live traces **and** dataset runs.
4. Push; push-to-main auto-deploys the dashboard (Vercel). No reviewer/Railway
   changes — this work is dashboard-only.
